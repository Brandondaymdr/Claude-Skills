## Core Principles

CLAUDE.md is loaded into context at the start of every session. Every line costs tokens and competes for Claude's attention. The goal is maximum signal, minimum noise.

**The 200-line budget.** Keep your root CLAUDE.md under 200 lines. Beyond this, Claude's compliance with instructions starts dropping. This isn't a hard limit — it's a quality threshold. If you need more, use progressive disclosure (skills, rules, references).

**The deletion test.** For every line, ask: "Would Claude make a mistake without this?" If Claude already does it correctly on its own, the instruction is noise. Remove it.

**Explain the why.** Instead of rigid MUST/NEVER rules, explain reasoning. Claude has good judgment — telling it *why* a convention exists lets it handle edge cases intelligently. Reserve emphasis (IMPORTANT, ALWAYS) for genuine foot-guns.

**CLAUDE.md is advisory (≈80% compliance).** For rules that must be followed 100% of the time (formatting, linting, security), use hooks instead. Hooks are deterministic scripts, not suggestions.

## What to Include

### 1. Project Overview (2-3 lines)
What this project is and what it does. One sentence of context goes a long way.

### 2. Architecture (5-10 lines)
Major components and how they connect. For non-trivial architectures, include a simple ASCII diagram. Mention the data flow — where requests come in, how they're processed, where data lives.

### 3. Tech Stack (3-5 lines)
Language, framework, database, hosting, and key libraries. Include version info when it matters (e.g., "Next.js 14 with App Router" vs just "Next.js").

### 4. Commands (5-10 lines)
Every command Claude needs to install, run, test, build, lint, and deploy. These should be copy-paste ready. Include flags that aren't obvious.

```markdown
## Commands
- `pnpm install` — install dependencies
- `pnpm dev` — start dev server (port 3000)
- `pnpm test` — run vitest
- `pnpm test:watch` — run vitest in watch mode
- `pnpm build` — production build
- `pnpm lint` — eslint + prettier check
- `pnpm lint:fix` — auto-fix lint issues
- `pnpm db:push` — push schema to Supabase
- `pnpm db:generate` — generate types from schema
```

### 5. Code Conventions (5-15 lines)
Only conventions that **differ from ecosystem defaults**. If you're writing standard TypeScript with standard ESLint rules, don't document that. Document the surprises:

```markdown
## Code Conventions
- Use `type` over `interface` for object types (project convention)
- All API routes return `{ data, error }` shape — never throw from API handlers
- Use Supabase RLS for authorization, not middleware checks
- File naming: kebab-case for files, PascalCase for components
- Imports: group by external → internal → relative, with blank lines between
```

### 6. Workflow (3-5 lines)
How work gets committed and shipped. Branch strategy, commit conventions, PR requirements.

### 7. Project-Specific Gotchas (3-10 lines)
Things that would trip Claude up without being told. These are the highest-value lines in CLAUDE.md:

```markdown
## Gotchas
- The `users` table has RLS enabled — all queries must use the service role key or go through authenticated client
- Don't modify `generated/` files directly — they're auto-generated by `pnpm db:generate`
- The Vercel build uses `pnpm` not `npm` — check `vercel.json` before adding build commands
- Tests use a separate `.env.test` file — don't mix test and dev env vars
```

### 8. Skills & References (2-5 lines)
Point to where deeper knowledge lives. This is the progressive disclosure mechanism:

```markdown
## References
- See `.claude/skills/api-design/` for REST API conventions
- See `docs/database-schema.md` for the full data model
- See `.claude/rules/` for path-specific coding rules
```

## What to Exclude

- **Standard language conventions** Claude already knows. Don't tell it to "use async/await" in JavaScript.
- **Linter rules** that are already in your config files. Claude reads those.
- **File-by-file descriptions** of the codebase. Claude can explore.
- **Long tutorials or explanations.** Move these to docs/ or skills.
- **Information that changes frequently.** It'll be wrong more often than right.
- **"Write clean code" / "Follow best practices"** — these are empty instructions.
- **Git history summaries.** Use `git log`.

## Progressive Disclosure

When CLAUDE.md grows past 200 lines, redistribute content:

| Content Type | Where It Goes | Why |
|---|---|---|
| Always-on conventions | CLAUDE.md | Every session needs these |
| Path-specific rules | `.claude/rules/` with path frontmatter | Only loads when working in those paths |
| Domain knowledge | `.claude/skills/` | Loads on demand when relevant |
| Deep reference docs | `docs/` or skill `references/` | Read explicitly when needed |
| One-off workflows | `.claude/commands/` | Invoked manually with `/project:name` |

### Using @imports

CLAUDE.md supports `@path/to/file` imports for content that's maintained elsewhere:

```markdown
## API Conventions
See @docs/api-conventions.md for the full API style guide.

## Git Workflow
Follow @docs/git-workflow.md for branching and PR conventions.
```

The imported file's content is included when CLAUDE.md loads, so use this sparingly — it still costs context.

## Multi-File Strategy

For monorepos or large projects, use multiple CLAUDE.md files:

```
project-root/
├── CLAUDE.md                  # Shared conventions, commands
├── packages/
│   ├── api/
│   │   └── CLAUDE.md          # API-specific conventions
│   ├── web/
│   │   └── CLAUDE.md          # Frontend-specific conventions
│   └── shared/
│       └── CLAUDE.md          # Shared library conventions
```

Claude automatically loads the root CLAUDE.md plus any CLAUDE.md files in the directory tree it's working in. Child files add context; they don't override the root.

## Maintenance

Treat CLAUDE.md as a living document:

- **After correcting Claude twice** on the same issue, add a rule.
- **After a project phase completes**, prune rules that no longer apply.
- **When a convention changes**, update the file immediately.
- **Periodically ask Claude** "What in CLAUDE.md is unclear or contradictory?" to find issues.
- **Tell Claude to update it** as you work: "Update CLAUDE.md with what we just decided about the error handling pattern."

## Examples

### Minimal (Solo side project)
```markdown
# MyApp

Personal expense tracker built with SvelteKit + Supabase.

## Commands
- `pnpm dev` — dev server on port 5173
- `pnpm test` — run vitest
- `pnpm build` — production build

## Stack
SvelteKit 2, Supabase (auth + database), Vercel deployment.

## Rules
- Use Supabase client from `$lib/supabase.ts`, never create new instances
- All database types are in `src/lib/types/database.ts` (auto-generated)
```

### Standard (Team web app)
```markdown
# Acme Dashboard

Internal analytics dashboard for the sales team.

## Architecture
Next.js App Router frontend → tRPC API layer → Supabase PostgreSQL.
Auth via Supabase with Google OAuth. Deployed on Vercel.

## Tech Stack
- Next.js 14 (App Router), TypeScript strict
- tRPC v11 for type-safe API
- Supabase (PostgreSQL + Auth + Realtime)
- Tailwind CSS + shadcn/ui components
- Vercel (preview deploys on PRs)

## Commands
- `pnpm dev` — dev server (port 3000)
- `pnpm test` — vitest
- `pnpm build` — production build
- `pnpm lint` — eslint + prettier
- `pnpm db:push` — push schema changes
- `pnpm db:generate` — regenerate types

## Code Conventions
- API routes: `app/api/trpc/[trpc]/route.ts` handles all tRPC
- Components: colocate with page in `_components/` directories
- Use `type` not `interface` for object types
- Errors: return `{ data: null, error: string }` from server actions

## Workflow
- Branch off `main`, PR required for merge
- Conventional commits (feat:, fix:, chore:)
- Run `pnpm lint && pnpm test` before committing

## Gotchas
- RLS is enabled on all tables — use `supabaseAdmin` for server-side queries
- The `generated/` directory is auto-generated — don't edit manually
- Vercel preview URLs use a different Supabase project (staging)

## References
- See `.claude/skills/` for API design conventions
- See `docs/database-schema.md` for full schema documentation
```

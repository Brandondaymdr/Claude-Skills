# Plaid API Authentication Guide

## Getting Credentials from Plaid Dashboard

### Step 1: Create a Plaid Account

1. Go to https://dashboard.plaid.com
2. Sign up for a free account (or log in if you have one)
3. Complete email verification

### Step 2: Create an Application

1. In the dashboard, navigate to **Team Settings** > **Applications**
2. Click **Create New Application**
3. Give your application a name (e.g., "ShoreStack Books")
4. Select the products you need (auth, transactions, identity, etc.)
5. Choose your country/region
6. Click **Create Application**

### Step 3: Retrieve Your Credentials

Your dashboard shows three sets of credentials for different environments:

**Sandbox Environment:**
- Client ID: `client_id_sandbox_xxxxx`
- Secret: `secret_sandbox_xxxxx`
- Base URL: `https://sandbox.plaid.com`

**Development Environment:**
- Client ID: `client_id_development_xxxxx`
- Secret: `secret_development_xxxxx`
- Base URL: `https://development.plaid.com`
- Requires test credentials from financial institutions

**Production Environment:**
- Client ID: `client_id_production_xxxxx`
- Secret: `secret_production_xxxxx`
- Base URL: `https://production.plaid.com`
- For live transactions with real credentials

## Header vs Body Authentication

### Headers (Recommended)

Headers are cleaner and more secure:

```javascript
const configuration = new Configuration({
  basePath: PlaidEnvironments.sandbox,
  baseOptions: {
    headers: {
      'PLAID-CLIENT-ID': process.env.PLAID_CLIENT_ID,
      'PLAID-SECRET': process.env.PLAID_SECRET,
    },
  },
});

const client = new PlaidApi(configuration);
const response = await client.accountsGet({ access_token });
```

### Request Body (Alternative)

Some older SDKs use request body authentication:

```javascript
const response = await client.accountsGet({
  access_token,
  client_id: process.env.PLAID_CLIENT_ID,
  secret: process.env.PLAID_SECRET,
});
```

**Note:** Plaid recommends using headers instead. Always use headers with modern SDKs.

## Token Lifecycle

Understanding the token flow is critical for secure integration:

### Stage 1: Link Token

Used to initialize the Plaid Link UI on the frontend.

**Characteristics:**
- Short-lived (10 minute expiration)
- Generated server-side
- Used in frontend JavaScript to initialize Link
- One-time use

**Generation:**
```javascript
const linkTokenResponse = await client.linkTokenCreate({
  user: { client_user_id: 'user-123' },
  client_name: 'My App',
  products: ['auth', 'transactions'],
  country_codes: ['US'],
  language: 'en',
});

const linkToken = linkTokenResponse.data.link_token;
const expiration = linkTokenResponse.data.expiration; // ISO timestamp
```

**Frontend usage:**
```javascript
// Pass linkToken to your frontend
const response = await fetch('/api/link-token', { method: 'GET' });
const { link_token } = await response.json();

// Initialize Plaid Link
const handler = Plaid.create({
  token: link_token,
  onSuccess: (public_token, metadata) => {
    // Send public_token to server for exchange
    sendToServer(public_token);
  },
  onExit: (err, metadata) => {
    // Handle user closing Link
  },
});

handler.open();
```

### Stage 2: Public Token

Returned from Plaid Link after user authenticates.

**Characteristics:**
- Single-use token
- Generated by frontend (from user authentication)
- Returned to your server
- Valid for ~30 minutes
- Must be immediately exchanged for access_token

**Exchange:**
```javascript
// Server receives public_token from client
const exchangeResponse = await client.itemPublicTokenExchange({
  public_token: publicTokenFromClient,
});

const accessToken = exchangeResponse.data.access_token;
const itemId = exchangeResponse.data.item_id;
```

### Stage 3: Access Token

Long-lived token used for all subsequent API calls.

**Characteristics:**
- Long-lived (valid until item is removed or credentials expire)
- Unique per item (per user connection)
- Must be stored securely in your database
- Used for all API calls after exchange
- Can expire if user's bank credentials are changed

**Storage best practice:**
```javascript
// Store in database, associated with user
await db.users.update(userId, {
  plaid_access_token: accessToken,
  plaid_item_id: itemId,
  plaid_connected_at: new Date(),
});
```

**Usage:**
```javascript
// Retrieve from database when needed
const user = await db.users.findById(userId);
const accessToken = user.plaid_access_token;

// Use in all subsequent calls
const accounts = await client.accountsGet({ access_token: accessToken });
```

## Token Storage Best Practices

### 1. Database Storage

Never store tokens in browser localStorage, cookies, or client-side.

```javascript
// Backend: Database schema example
const PlaidTokenSchema = {
  user_id: String,
  access_token: String, // Encrypted
  item_id: String,
  institution_id: String,
  connected_at: Date,
  last_synced: Date,
  status: String, // 'active', 'expired', 'error'
};

// Store encrypted
const encrypted = await encryptToken(accessToken);
await db.plaidTokens.create({
  user_id: userId,
  access_token: encrypted,
  item_id: itemId,
  status: 'active',
});
```

### 2. Encryption

Encrypt tokens at rest in the database:

```javascript
const crypto = require('crypto');

function encryptToken(token, encryptionKey) {
  const iv = crypto.randomBytes(16);
  const cipher = crypto.createCipheriv(
    'aes-256-cbc',
    Buffer.from(encryptionKey),
    iv
  );
  let encrypted = cipher.update(token, 'utf8', 'hex');
  encrypted += cipher.final('hex');
  return `${iv.toString('hex')}:${encrypted}`;
}

function decryptToken(encrypted, encryptionKey) {
  const [iv, cipher_text] = encrypted.split(':');
  const decipher = crypto.createDecipheriv(
    'aes-256-cbc',
    Buffer.from(encryptionKey),
    Buffer.from(iv, 'hex')
  );
  let decrypted = decipher.update(cipher_text, 'hex', 'utf8');
  decrypted += decipher.final('utf8');
  return decrypted;
}
```

### 3. Access Control

Only retrieve tokens for the authenticated user:

```javascript
// Middleware: Verify user is accessing their own token
app.get('/api/plaid/accounts', authenticateUser, async (req, res) => {
  const userId = req.user.id;

  // Retrieve token for THIS user only
  const plaidData = await db.plaidTokens.findOne({ user_id: userId });

  if (!plaidData) {
    return res.status(404).json({ error: 'Not connected to Plaid' });
  }

  const accessToken = await decryptToken(
    plaidData.access_token,
    process.env.ENCRYPTION_KEY
  );

  const accounts = await client.accountsGet({ access_token: accessToken });
  res.json(accounts.data);
});
```

### 4. Secure Transmission

Always use HTTPS and never pass tokens in URLs:

```javascript
// GOOD: Token in request body with HTTPS
fetch('https://app.example.com/api/accounts', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ /* no tokens */ }),
});

// BAD: Token in URL (visible in logs and referrers)
fetch(`https://app.example.com/api/accounts?token=${accessToken}`);

// BAD: Token in unencrypted HTTP
fetch('http://app.example.com/api/accounts', {
  body: JSON.stringify({ access_token: accessToken }),
});
```

### 5. Token Rotation

Monitor and rotate tokens periodically:

```javascript
// Check for expired/error status
app.post('/api/plaid/sync', async (req, res) => {
  const userId = req.user.id;
  const plaidData = await db.plaidTokens.findOne({ user_id: userId });

  if (plaidData.status === 'expired' || plaidData.status === 'error') {
    // Ask user to reconnect
    return res.status(400).json({
      error: 'Plaid connection expired',
      action: 'reconnect',
    });
  }

  // Use token
  const accessToken = await decryptToken(
    plaidData.access_token,
    process.env.ENCRYPTION_KEY
  );

  try {
    const response = await client.accountsGet({ access_token: accessToken });
    // Update last_synced
    await db.plaidTokens.updateOne(
      { user_id: userId },
      { last_synced: new Date(), status: 'active' }
    );
    res.json(response.data);
  } catch (error) {
    if (error.response?.data?.error_code === 'ITEM_LOGIN_REQUIRED') {
      // Mark as expired
      await db.plaidTokens.updateOne(
        { user_id: userId },
        { status: 'expired' }
      );
    }
    throw error;
  }
});
```

## Processor Tokens

Processor tokens are special tokens for payment partners.

### Overview

Instead of giving a payment processor your access_token, generate a processor token:

```
User -> Your App (holds access_token) -> Payment Processor (has processor_token)
```

### Creation

```javascript
// Create once per account per processor
const processorResponse = await client.processorTokenCreate({
  access_token: userAccessToken,
  account_id: selectedAccountId,
  processor: 'dwolla', // or 'stripe', 'fiserv', etc.
});

const processorToken = processorResponse.data.processor_token;

// Store and give to payment processor
await paymentProcessor.setToken(userId, processorToken);
```

### Supported Processors

| Processor | Key | Use Case |
|-----------|-----|----------|
| Dwolla | `dwolla` | ACH transfers |
| Stripe | `stripe` | Payments |
| Fiserv | `fiserv` | Banking services |
| Finicity | `finicity` | Lending decisions |
| Check | `check` | Check writing |

### Benefits

- Payment processor never sees your access_token
- Tokens are processor-specific and can't be used elsewhere
- Easier to revoke: just regenerate a new processor token
- Better security isolation

## Security Checklist

- [ ] Credentials stored in environment variables, not in code
- [ ] Each environment (sandbox/dev/prod) uses separate credentials
- [ ] Access tokens encrypted at rest in database
- [ ] Access tokens never logged or sent to client
- [ ] HTTPS required for all token transmission
- [ ] Tokens only used server-side
- [ ] Rate limiting implemented to prevent abuse
- [ ] Secrets rotated regularly (monthly recommended)
- [ ] Access token expiration monitored
- [ ] Webhook signatures verified (if using webhooks)
- [ ] Database backups encrypted
- [ ] Audit logging for token access

## Environment Setup Example

### .env file

```
# Sandbox
PLAID_CLIENT_ID_SANDBOX=client_id_sandbox_xxxxx
PLAID_SECRET_SANDBOX=secret_sandbox_xxxxx

# Development
PLAID_CLIENT_ID_DEVELOPMENT=client_id_development_xxxxx
PLAID_SECRET_DEVELOPMENT=secret_development_xxxxx

# Production
PLAID_CLIENT_ID_PRODUCTION=client_id_production_xxxxx
PLAID_SECRET_PRODUCTION=secret_production_xxxxx

# Current environment
NODE_ENV=sandbox
ENCRYPTION_KEY=your_256_bit_hex_key_here
```

### Node.js Setup

```javascript
require('dotenv').config();

const { Configuration, PlaidApi, PlaidEnvironments } = require('plaid');

const getPlaidClient = () => {
  const env = process.env.NODE_ENV || 'sandbox';
  const clientIdKey = `PLAID_CLIENT_ID_${env.toUpperCase()}`;
  const secretKey = `PLAID_SECRET_${env.toUpperCase()}`;

  const envMap = {
    sandbox: PlaidEnvironments.sandbox,
    development: PlaidEnvironments.development,
    production: PlaidEnvironments.production,
  };

  const configuration = new Configuration({
    basePath: envMap[env],
    baseOptions: {
      headers: {
        'PLAID-CLIENT-ID': process.env[clientIdKey],
        'PLAID-SECRET': process.env[secretKey],
      },
    },
  });

  return new PlaidApi(configuration);
};

module.exports = { getPlaidClient };
```

### Python Setup

```python
import os
from dotenv import load_dotenv
import plaid

load_dotenv()

def get_plaid_client():
    env = os.getenv('NODE_ENV', 'sandbox')
    client_id_key = f'PLAID_CLIENT_ID_{env.upper()}'
    secret_key = f'PLAID_SECRET_{env.upper()}'

    env_map = {
        'sandbox': plaid.ApiClient.Environment.SANDBOX,
        'development': plaid.ApiClient.Environment.DEVELOPMENT,
        'production': plaid.ApiClient.Environment.PRODUCTION,
    }

    configuration = plaid.Configuration(
        host=env_map[env],
        api_key=os.getenv(client_id_key),
        api_key_prefix='',
        api_key_2=os.getenv(secret_key),
        api_key_prefix_2='',
    )

    api_client = plaid.ApiClient(configuration)
    return plaid.PlaidApi(api_client)
```

## Credential Rotation

Rotate credentials regularly for security:

### In Plaid Dashboard

1. Go to **Team Settings** > **Applications**
2. Select your application
3. Click **Generate New Credentials** (or **Regenerate Secret**)
4. Update your environment variables
5. Restart your application

### Timeline

1. Generate new credentials
2. Update one application instance at a time
3. Monitor logs for errors
4. After all instances updated and stable (24-48 hours), delete old credentials

```javascript
// Graceful transition with both old and new credentials
const clientIds = [
  process.env.PLAID_CLIENT_ID_NEW,
  process.env.PLAID_CLIENT_ID_OLD, // Fallback
];

const getWorkingClient = () => {
  // Try new first, fall back to old
  for (const clientId of clientIds) {
    const configuration = new Configuration({
      baseOptions: { headers: { 'PLAID-CLIENT-ID': clientId } },
    });
    return new PlaidApi(configuration);
  }
};
```
